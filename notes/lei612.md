今天整理部分基础概念：**LLM**：大语言模型本质是基于Transformer架构的自回归神经网络，通过在海量文本上学习token间的条件概率分布，在给定上文时逐次预测下一个最可能的token来生成自然语言（这里的一个token是通过分词器从文本字符串里拆分的，常用分词器有Byte Pair Encoding（BPE）以及一些变体），常见误区是以为模型有逻辑推理能力，实际它只是做模式匹配和概率采样，所以内容不是完全正确的。

**Prompt**：向LLM输入的文本指令或上下文，通过设计特定的角色、格式要求或示例来引导模型的条件概率分布偏向期望输出，本质是约束生成空间的先验条件。例如，Prompt增加了强制的前缀 token，模型基于这些前缀在训练数据中匹配到相似模式，从而调整 token 概率分布，但是不是越长越精确，冗余信息也会参与条件概率计算，会引入噪声稀释注意力。

**Context Window**：LLM一次能处理的token总数限制（包含输入和输出），模型内部使用位置编码和注意力机制，超出窗口的文本会被物理截断，这里的物理截断是直接在输入模型前丢弃多余部分，被丢弃的部分根本不会进入模型，所以不存在输入了注意力权重为0的情况。

**Workflow**：将多个LLM调用、规则判断和人工操作按固定DAG（有向无环图）编排成的自动化流水线，每一步的输出作为下一步的输入（这里用DAG是因为真实流程不是单一路径，所以不能直接用线性链，而且有步骤依赖，所以“有向”,工作流如果循环会无限执行下去，所以“无环”），无运行时分支决策。 这里我让deepseek给我生成了一个例子，但其实我这里没明白Agent 行为怎么穿插在这个流程里面的，目前有个比较浅的理解是：要么作为 Workflow 中的一个智能节点（封装动态决策），要么作为主控制器动态调用 Workflow 作为工具（Workflow 降级为可复用的子程序）。Agent 负责“做什么、什么时候做”，Workflow 负责“怎么做“

1.  **用户输入**：例如“我的订单12345怎么还没发货？”
    
2.  **意图分类（LLM 调用）**：使用 prompt 强制输出三类之一：`订单查询`、`物流查询`、`投诉`。模型输出 `订单查询`。
    
3.  **运行时分支决策（静态）**：工作流引擎检查分类结果：
    
    -   如果是 `订单查询` → 走 **C→F→I** 路径
        
    -   如果是 `物流查询` → 走 **D→G→I** 路径
        
    -   如果是 `投诉` → 走 **E→H→I** 路径
        
    
    这里的“决策”仅仅是查表跳转，**没有** LLM 去思考“咦，用户好像情绪激动，我是不是应该先安慰一下再转人工？”——后者属于 Agent 行为。
    
4.  **调用订单 API**（C 节点）：固定代码调用 `GET /order?orderId=12345`，返回订单状态。
    
5.  **LLM 生成回复**（F 节点）：预设模板 prompt：“根据订单状态：{订单状态}，生成一句礼貌的回复。” 输出“您的订单已发货，预计三天内到达。”
    
6.  **发送回复**（I 节点）：将文本发回用户。
    

**Agent**：一个包含 LLM、工具、记忆、循环控制在内的完整认知架构，能执行“感知-规划-行动”的智能体系统，它动态拆解目标、选择工具、执行动作并观察结果，根据反馈自动修正下一步计划，直到任务完成。例如：Agent收到“帮我订周五去上海的机票”，它会：查询航班→发现价格高→尝试改周四→对比选项→最终确认订单。这里误区是Agent只能用LLM做大脑，实际上完全可以嵌入不同类型的大模型，甚至同时使用多个大模型，一个例子是：**LangChain** 的MultiModelAgent ：可以同时绑定一个 GPT-4（推理）和一个 CLIP（图像分类）。当用户传图时，Agent 自动调用 CLIP 获得标签，再交给 GPT-4 生成回答

**Tool Use**：LLM在生成过程中输出特定格式（如JSON/函数调用）来请求外部系统执行函数（如API、数据库、代码解释器），系统将执行结果拼接到上下文后模型继续推理，本质是让模型学会调用外部能力的“对话协议”。用户问“北京天气”，模型输出`{“function":"get_weather","city":"北京"}`，系统调用天气API后把“晴25°C”送回模型再生成回答。这里不是模型调用的接口，模型只输出一个json，然后**Tool Executor**接收这个json，解析并提取关键参数，然后去一个预先注册好的函数字典里查找对应的可调用函数。
