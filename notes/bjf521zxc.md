# Web3 钱包基础笔记

## 一个最重要的认知转变

传统 App 的账号是平台给你的——忘记密码？找客服。账号被封？申诉。**平台是管理员。**

Web3 钱包完全相反：**你的私钥 = 你的账户**，没有管理员，没有找回密码。平台只是请求连接的客人，不是房东。

> 类比：钱包是你的身份证 + 印章 + 保险箱钥匙合为一体。App 能看你的名字，但不能开你的保险箱——除非你把钥匙交出去。

* * *

## 三类动作，风险完全不同

很多人把"连接钱包"理解成"登录了就能操作一切"，这是危险误区。

| 动作 | 类比 | 实际发生什么 | 风险 |
| --- | --- | --- | --- |
| 连接钱包 | 递名片 | App 读取你的地址和当前网络 | 极低，不动资产 |
| 签名消息 | 签一份声明 | 证明你控制该地址，常用于登录/授权 | 中，要看签的是什么 |
| 发送交易 | 盖章转账 | 真正改变链上状态，消耗 gas | 高，不可随意撤回 |

**设计原则**：这三类动作的页面文案、按钮状态、确认提示必须区别对待。"连接"和"确认转账"用同一个样式按钮，是严重的 UX 失职。

* * *

## 账户类型：EOA vs 智能账户

**EOA（普通钱包账户）** 就像一把传统机械锁——简单可靠，但钥匙（私钥）一丢就完了，没有备用方案，不能设定"只允许小额自动扣款"这类细粒度规则。

**Smart Account（账户抽象）** 像换了一把智能门锁——可以设置指纹、临时密码、限额规则，还能让别人代付门票（gas）。但目前兼容性和复杂度更高。

> 大多数用户现在用的都是 EOA。理解它的限制，是理解为什么 UX 经常被打断（签名弹窗、网络切换、gas 不足）的前提。

* * *

## 助记词：一个永远不该出现在表单里的东西

助记词是用来**在钱包 App 内部**恢复账户的，不是输入框里的验证码。

> 类比：助记词是保险箱的主密码。你去银行办业务，柜员绝不会要求你当众说出密码。任何要求你"输入助记词"的网页/客服/AI，都在试图撬保险箱。

**正确流程**：用户在钱包 App 里恢复账户 → dApp 请求连接 → 用户授权。中间**不经过**任何第三方表单。

* * *

## 交易的生命周期（按钮按下去之后发生了什么）

很多人以为"点击确认 = 完成"。实际上交易要经历：

```
用户点击按钮
    → 钱包弹出确认窗口（用户可以在这里拒绝）
    → 用户签名 → 广播到网络
    → 矿工/验证者打包（可能等待）
    → 上链执行（可能 revert）
    → 最终状态确认
```

每一步都可能失败。前端**至少**要处理：等待确认中 / 用户拒绝 / 广播成功等待打包 / 成功 / 失败 revert / 长时间 pending。

> 类比：发一张支票，不是签了字就收到钱——还要过结算、入账、到账。"签字"只是起点。

* * *

## Gas：邮票，不是小费

Gas 是你付给网络的**执行费**，不是可选的打赏。每一笔交易都要消耗。

关键反直觉点：**交易失败了，gas 仍然可能被消耗**（因为网络已经执行了，只是执行结果是失败）。

产品里不能只写"确认"，至少要让用户看到：费用大概多少、用什么资产付、失败是否还扣费。

* * *

## Explorer：链上的"查账单"入口

区块浏览器（Etherscan 等）是任何人都能用来**核实链上事实**的工具。

看一笔交易时关注：Status（成功/失败）、From/To（谁发给谁）、Method（调了哪个合约方法）、Token Transfers（有没有 token 转移）、Gas Used。

> AI 产品引用链上信息时，给出 explorer 链接比只给结论重要得多——让用户自己验证，而不是"信我"。

* * *

## AI Agent 碰到钱包边界

AI 可以帮你**理解**交易参数、**检查**风险、**准备**操作计划，但签名和授权不能交给模型自动完成。

合理分工：**AI 做参谋，钱包做审批，合约做执行约束。** 效率和控制权都保留。

````markdown
## 一、三件套的逻辑关系

```
随机数(256bit)
    │  椭圆曲线点乘（secp256k1）
    ▼
  公钥
    │  Keccak-256 哈希 → 取后 20 字节
    ▼
  地址（你的链上身份）
```

| 概念 | 类比 | 关键性质 |
|------|------|----------|
| 私钥 | 印章原件 | 单向生成公钥，不可逆推 |
| 公钥 | 印章图样 | 可公开，用于验证签名 |
| 地址 | 银行卡号 | 公钥哈希截取，20 字节 |

> **认知纠正**：公钥 ≠ 地址。地址是公钥经过 Keccak-256 再截取后 20 字节的结果。

---

## 二、哈希函数：信息的"指纹机器"

不要把哈希理解为加密——它**不可逆**，更像指纹。

**Keccak-256 的四个关键性质：**

1. **确定性** — 同样输入必得同样输出
2. **雪崩效应** — 改变 1 bit，输出面目全非
3. **不可逆** — 无法从输出反推输入
4. **抗碰撞** — 极难找到两个不同输入产生相同哈希

**在 Web3 里的用途：**
- 把任意长度的交易数据压缩成 32 字节摘要
- 对摘要签名（而非对整笔数据签名），效率高，安全性一致
- 生成地址、构建 Merkle Tree、区块哈希链接

---

## 三、非对称加密：椭圆曲线凭什么安全？

以太坊使用 **secp256k1** 椭圆曲线：

```
私钥 k（随机整数）
公钥 K = k × G     ← G 是曲线的"生成点"，公开常数
```

**为什么单向？**
从 `k` 计算 `K` 是重复做椭圆曲线加法，容易；
从 `K` 反推 `k` 是"椭圆曲线离散对数问题"，目前没有高效算法，
即使用现有最快的计算机穷举，也需要远超宇宙年龄的时间。

> **关键认知**：私钥 = 所有权本身，不是"密码"。没有"忘记密码重置"——丢了就是丢了。

---

## 四、数字签名：ECDSA 的工作流程

### 签名（发送方）

```
① 构造交易对象（from, to, value, nonce, gasPrice, gasLimit）
② 对交易数据做 Keccak-256 → 得到 32 字节摘要 h
③ 用私钥 k 对 h 做 ECDSA 签名 → 产生 (r, s, v)
④ 把原始交易 + 签名 (r, s, v) 广播到网络
```

### 验证（节点）

```
① 收到原始交易 + (r, s, v)
② 重新计算交易哈希 h
③ ecrecover(h, r, s, v) → 恢复出公钥 → 算出地址
④ 对比 from 字段：地址匹配 → 合法
⑤ 检查 nonce、余额 → 打包进区块
```

**签名三个值的含义：**

| 值 | 含义 |
|----|------|
| `r` | 签名时随机点的 x 坐标 |
| `s` | 签名证明值（与私钥、r、消息哈希相关） |
| `v` | 恢复标识符（27 或 28），用于 ecrecover 确定唯一公钥 |

> **关键认知**：签名不暴露私钥本身。验证者只需公钥，无需任何中间人参与。

---

## 五、一笔交易的完整字段

```json
{
  "from":     "0x742d35Cc...",   // 发送方地址（从签名恢复，非显式信任）
  "to":       "0xAbCd1234...",   // 接收方地址
  "value":    "1.5 ETH",         // 转账金额
  "nonce":    42,                // 防重放攻击（每笔交易唯一递增）
  "gasPrice": "25 Gwei",         // 每单位 Gas 的出价
  "gasLimit": 21000,             // 最多消耗多少 Gas
  "chainId":  1,                 // 防跨链重放（EIP-155）
  "r": "0x9f71...",
  "s": "0x4a5b...",
  "v": "0x1b"
}
```

**Gas 费计算：**
```
手续费 = gasPrice × 实际消耗 Gas
最大花费 = gasPrice × gasLimit
普通转账固定消耗 = 21,000 Gas
```

## 六、常见认知陷阱

| 误解 | 正确理解 |
|------|----------|
| 公钥 = 地址 | 公钥经 Keccak-256 再取后 20 字节才是地址 |
| 签名 = 加密内容 | 签名不加密，只证明"是私钥持有者发出的" |
| 私钥泄露可以更换 | 私钥与地址绑定，只能弃用该地址 |
| 链上交易内容保密 | 链上数据全部公开，匿名靠地址本身，不靠加密 |
| nonce 可以随便填 | nonce 必须严格递增，否则交易被拒绝或可被重放 |
````

    embedding（语义表征）
    
    transformer（核心架构）
    
3.  风险管理：幻觉
    
    成因：模型是基于概率输出而不是剋性能事实，这些内容无法验证
    
    风险：在执行能力系统中，错误参数、错误解释都会带来危险
    
    对策：将模型接入外部检验而不是只依赖提示词优化

    用于理解和生成层，将用户目标转为计划，解释复杂链上数据。同时还需要协同架构（数据层+执行层+安全层）
