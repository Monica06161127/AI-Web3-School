# Web3 運行原理：從第一性原理出發

當用戶在 Web3 錢包點擊「Confirm（確認）」的那一刻，整個區塊鏈系統便圍繞著三個核心問題開始運轉：你是誰？、你要做什麼？、為什麼相信這個結果？。以下是底層系統的完整解構。

## 一、 身份認證與個人主權：錢包與密碼學

### 1\. 助記詞、私鑰與地址的衍生關係

-   私鑰（Private Key）：控制帳戶資產的唯一鑰匙，本質上是一串極長的隨機亂數。
    
-   助記詞（Mnemonic Phrase）：由於私鑰過長、人類難以記憶，系統透過演算法將其轉換為一組容易記錄的單詞。一組助記詞可以透過不同的「派生路徑」，衍生管理無數組私鑰（一對多關係）。
    
-   錢包地址（Address）：私鑰透過非對稱加密演算法計算出公鑰。為了便於傳播與識別，以太坊會截取公鑰哈希值的最後 20 個字節（共 40 個字元）作為公開的錢包地址。由於長度足夠，哈希碰撞（撞衫）的機率極低。
    

### 2\. 錢包的本質與個人主權

-   資產不在錢包裡：Web3 錢包（如 MetaMask、Rabby）本質上只是一個私鑰管理與簽名工具。用戶的資產與帳本狀態完全儲存在區塊鏈網絡上。
    
-   無許可（Permissionless）創立：用戶生成錢包不需要向銀行或政府申請，也無需提供身份證明（KYC）。只要擁有私鑰，就擁有資產的絕對控制權，這是數字個人主權的起點。
    
-   安全警示：私鑰一旦洩露（如遭受釣魚攻擊、設備中木馬、截圖誤傳），資產將無法追回，因為區塊鏈只認簽名不認人。
    

## 二、 意圖授權：交易與數字簽名

### 1\. 交易（Transaction）的本質

在 Web3 中，交易不單指貨幣轉帳。它的本質是「一段授權區塊鏈網絡執行特定行為的加密數據」（包含轉帳、合約調用、參與投票或鑄造 NFT 等）。

### 2\. 核心數據欄位

-   行動內容：明確定義資金去向或調用的智能合約方法。
    
-   Nonce：發送帳戶的交易連續序號。每成功執行一筆交易，該值便加 1。其核心作用是防止「重放攻擊」，確保同一筆廣播出去的交易不會被惡意重複執行。
    
-   Gas 費（手續費）：用戶佔用區塊鏈網絡運算與存儲資源必須支付的代價。它能有效遏止惡意垃圾交易癱瘓網絡。Gas 費分為基礎費（直接銷毀以實現代幣通縮）與優先費（用以激勵驗證者優先打包）。
    

### 3\. 簽名與驗證流（ECRecover 演算法）

1.  在地簽名：用戶的錢包用私鑰結合交易數據，在本地計算生成一段數字簽名（數據一有微調，簽名就會完全改變）。
    
2.  廣播數據：錢包將「原始交易訊息」、「聲稱的發送方地址」與「數字簽名」三項數據廣播至網絡。
    
3.  還原驗證：區塊鏈節點收到後，運行 ECRecover 演算法，將「原始訊息 + 簽名」反推計算出簽名者真實地址，並與聲稱的發送方進行對比。兩者一致，則執行交易；遭篡改則直接拒絕。
    

## 三、 信任網絡：網絡傳播與共識機制

### 1\. 交易的生命週期

錢包在地簽名 ──> RPC 節點 ──> 內存池（Mempool）排隊 ──> 構建者（Builder）排序打包 ──> 驗證者（Validator）共識驗證 ──> 區塊鏈最終確認。

-   RPC 節點：區塊鏈是去中心化的 P2P 網絡，傳統 Web 網頁無法直接與其溝通。RPC（遠程過程調用）充當橋樑，提供傳統 HTTP 或 WebSocket 介面，方便網頁應用與錢包將交易送入區塊鏈。
    
-   內存池（Mempool）：全球交易蜂擁而至時，尚未被打包的交易會在節點內存中排隊等待。
    

### 2\. 共識機制（Consensus Mechanism）

共識機制是為了解決「在互不相識、缺乏信任的去中心化網絡中，如何安全記帳」的問題。

-   POW（工作量證明）：如比特幣。節點盲目碰撞哈希值解題，勝出者獲得記帳權。極度消耗電能，但去中心化安全性極高。
    
-   POS（權益證明）：如現行以太坊。驗證者需要質押 32 枚 ETH 才能建立節點。系統隨機挑選驗證者委員會進行記帳與見證。驗證者若試圖篡改數據或擅離職守，將面臨質押款被沒收（Slashing）的財務懲罰。
    

### 3\. 區塊鏈的最終性（Finality）

每個新區塊都會包含前一個區塊的哈希值（Parent Hash），環環相扣形成鏈條。剛產生的區塊存在被分叉或攻擊替換的微小風險（稱為 Unfinalized），隨著後續區塊不斷覆蓋，當網絡達到共識確認後（以太坊通常需要驗證約兩個時段），區塊便會進入 Finalized（最終確定） 狀態，此時資產狀態便幾乎不可逆轉。

## 四、 規則與演進：智能合約與協議升級

### 1\. 智能合約與社會學意義（Code is Law）

智能合約是以字節碼（Bytecode）形式寫在鏈上的程序，由 EVM（以太坊虛擬機）負責執行。

-   信任轉移：它將傳統商業中對「律師、政府、中介、法律合約」的信任，轉移為對「不可篡改之代碼、數學與共識」的信任。規則公開透明、強制執行，大幅降低了跨國協作與金融流轉的磨損成本。
    

### 2\. 智能合約如何實現「升級」

寫在鏈上的合約代碼一經部署便無法修改，為了解決 Bug 或進行功能擴展，業界引入了代理合約（Proxy Contract）模式。

-   用戶交互的永遠是「代理合約」的固定地址。
    
-   代理合約內部儲存了一個「指針」，指向真正負責運行的「邏輯合約」。
    
-   當需要升級時，開發者部署一個全新版本的邏輯合約，並更新代理合約中的指針指向。用戶端無需更改任何設定，即可完成功能升級。
    

### 3\. 底層協議層的去中心化升級

區塊鏈網絡本身的升級（如硬分叉、改變區塊數據結構）是一個去中心化的社會治理過程：

-   提案經論壇與核心開發者會議討論，形成 EIP/ERC 標準文檔。
    
-   全球不同團隊使用不同程式語言（如 Go、Rust）開發的多客戶端軟體（如 Geth、Nethermind 等）各自實施代碼，並在約定的區塊高度同步開啟新功能旗標。
    
-   多客戶端機制確保了當單一軟體出現漏洞（Bug）時，其餘客戶端仍能支撐網絡暢通，避免全網癱瘓。
    

## 五、 結論：Web3 的多學科交織

Web3 並非單純的資訊技術，而是三門學科的交叉結晶：

1.  密碼學（Cryptography）：利用哈希演算法、非對稱加密與簽名，保障數字資產的所有權與身份防偽。
    
2.  經濟學（Economics）：利用質押激勵、Gas 費處置與懲罰機制，協調全球陌生參與者
    
    的利益動機。
    
3.  社會學（Sociology）：利用去中心化治理、開源文化與社會共識，決定網絡的演進規則與主權分配。

### 一、 AI 的進化與 AI Agent 核心機制

-   **AI 的進化階段**：
    
    -   **早期（如 ChatGPT 剛推出時）**：以問答式交互為主，本質上像更智能的搜尋引擎，主要做解釋代碼、總結文章、寫文案等。
        
    -   **現階段（AI Agent 時代）**：AI 開始能執行實際任務，走向自動化編程與通用任務處理，從單純的「聊天助手」轉變為「行動智能體」。
        
-   **AI Agent 的核心運作機制**：
    
    用戶給予提示詞 -> 理解任務目標 -> 使用外部工具執行 -> 觀察並檢查執行結果 -> 若有錯誤進行**自我自動修復**，形成一個閉環。
    
-   **與一般聊天機器人的本質區別**：
    
    聊天機器人主要被動回答問題；而 Agent（智能體）則能理解人類語言背後的「目標」，主動調用工具去「替你完成事情」。
    

### 二、 市面上的 AI 生態五大陣營

講師將目前市面上的 AI 工具劃分為以下五大類別：

1.  **聊天型 AI**：如 ChatGPT、Claude、DeepSeek 等。
    
2.  **AI 編程助手/IDE**：如 Cursor、Windsurf、Copilot，這類工具多基於 VS Code 二次開發，內置 Agent 進行自動化代碼編寫。
    
3.  **終端型工具（CLI）**：針對開發者優化的命令行 Agent，如 `open-coder`、`codesk` 等。
    
4.  **模型平台**：如 OpenRouter 或魔搭社區，匯集並託管多種大模型供開發者對接調用。
    
5.  **通用助手底座（如 OpenHands、Hermes Agent）**：這類工具連接了大模型、本地工具、記憶系統與消息平台，就像一個小型「賈維斯」（Jarvis），只要給予授權，就能在系統後台幫你做很多事。
    

### 三、 Hermes Agent 的核心優勢與能力

影片重點推薦並教學 **Hermes Agent**，它具備以下四大超能力：

-   **多模態接入與工具調用**。
    
-   **長期記憶機制**：具備記憶持久化功能（會寫入 `memory.md` 檔案），且優化做得很好，磁碟占用不會隨著時間無限膨脹。
    
-   **技能（Skills）沉澱**：會把日常使用的固定任務轉化為模板化的「工作流」。當遇到類似任務時，會自動調用已有的 Skill 執行。
    
-   **對通訊軟體友好**：小白可能不會用複雜的終端命令行，但 Hermes 支持對接**微信（WeChat）、Telegram、飛書**等通訊軟體，讓用戶直接用聊天方式來操控電腦執行任務。
    

### 四、 Windows（WSL2）環境安裝與配置要點

講師在影片中實機演示了環境搭建的步驟：

-   **環境建議**：強烈建議 Windows 用戶安裝 **WSL2（Ubuntu）** 子系統來運行。Hermes 運行對本機硬體要求不高，實測 3-4GB 記憶體即可流暢運行。
    
-   **一鍵安裝**：到 GitHub 下載 Hermes 倉庫，執行官方的一鍵安裝指令，系統會自動下載配置 Node.js、Python 等環境依赖與依賴項。
    
-   **基本配置（Setup）**：
    
    -   使用 `quick setup` 配置 Provider（大模型提供者，如 Claude 3.5）。
        
    -   支持配置付費的 API Key 或是綁定已購買的 Claude Pro 帳號（透過瀏覽器 Token 授權）。
        
    -   配置 Gateway（消息網關），講師現場演示了掃碼對接微信的過程。
        

### 五、 講師的 Hermes 私人應用場景分享

講師展示了他自己的 Telegram/微信 Agent 分組與日常實用案例：

1.  **General（私人助理）**：處理日常雜事。例如讓 Agent 每天早上定時抓取最新的 Web3 產業新聞，自動整理成一份簡報發給自己。
    
2.  **Code（編程輔助）**：進行技術概念驗證。講師不建議直接讓它寫大型工程代碼，更適合當作邏輯構思的黑客松助手。
    
3.  **量化交易（Quant）**：講師說明他不依賴 AI 來做量化策略（因為開源策略多已失效），而是把個人的交易習慣、標的、風控準則交給 Agent，讓 Agent 當作助手幫忙記錄交易行為與生成報告。
    
4.  **學習助手（Learning Agent）**：官方提供了一個學習智能體的啟動入口。複製指令發給 Hermes 後，Agent 會引導用戶輸入「用戶畫像」（包含目前的 Web3 技術水平、AI 基礎、學習目標與今日可用時間），藉此為用戶客製化當天的學習計劃與進度評估。
    

### 六、 安全與防護設置建議

-   **單次授權**：在通訊軟體上操控 Agent 時，如果涉及本地系統的高風險操作（Dangerous command），系統會彈出預警。強烈建議新手選擇「單次授權」，看清楚 Agent 想要執行的指令再放行。
    
-   **白名單與喚醒詞**：建議在配置中開啟白名單（限定只有自己或特定好友能使用）與個人化的喚醒詞（例如：小黑幫我…），避免 Agent 在群組中誤觸或胡亂回應不相關的聊天訊息。

### 核心背景與基礎概念

什麼是 MPC（多方安全計算）？

-   傳統私鑰：如同開啟金庫的唯一實體鑰匙，一旦遺失或被竊，資產便徹底不保。
    
-   MPC 技術：將鑰匙「打碎成多個碎片（私鑰分片）」，由不同成員（如用戶手機、雲端伺服器）各自保管。
    
-   共同簽名：轉帳時，各方拿出各自的碎片進行「密碼學運算」，在不拼湊出完整鑰匙的前提下，共同完成交易簽名。
    

### BitForge 漏洞的本質

-   漏洞名稱：BitForge
    
-   受影響對象：幾乎所有主流開源的 GG18 與 GG20 MPC 協議實現。
    
-   問題根源：MPC 協議的安全建立在特定的數學假設（Paillier 同態加密）之上。
    
-   該漏洞利用了系統未嚴格檢查參與者金鑰合法性的漏洞，導致「安全地基」崩塌。
    

### 駭客攻擊步驟拆解

駭客若想竊取其他參與者（例如雲端伺服器）的私鑰碎片，會經過以下三個階段：

1.  第一步：準備階段（構造非法金鑰） 正常做法：使用非常巨大且複雜的兩個大質數 p, q 來生成數學保險箱的公鑰 N = p 乘 q。 駭客作弊：故意不用大質數，而是用很多「很小的質數」拼湊成公鑰 N = p1 乘 p2 乘 ... 乘 p16 乘 q。 結果：由於原協議只檢查了保險箱是否「無平方因子（Square-Free）」，駭客成功騙過系統混了進去。
    
2.  第二步：攻擊階段（在轉帳簽名時作弊） 駭客故意發起交易，並在簽名運算（MtA 協議）中，用特殊設計的數值代替隨機數。 正常的「零知識證明（Range Proof）」本該擋下這種惡意數值，但駭客利用第一步那個「偷工減料的保險箱」，成功偽造了假的證明，騙過其他參與者。 駭客在順利完成簽名的過程中，偷偷記錄下每次運算留下的數學線索。
    
3.  第三步：收尾階段（拼湊真相） 駭客只需要重複發起約 16 次轉帳簽名，就能收集到 16 個不同小因子對應的線索。 最終利用數學上的「中國剩餘定理（CRT）」，將這 16 個線索進行拼湊，就能精準推導出被攻擊方的私鑰碎片！ 最終後果：駭客擁有自己的碎片，又偷到了別人的碎片，即可合成完整私鑰，徹底掌控錢包。
    

### 防禦與修復方案（補上漏洞） 安全團隊（Fireblocks 與 Safeheron）提出的修復方案是從源頭掐斷攻擊，也就是在建立錢包時，強制進行「零件硬度檢測」。

### 在協議中額外添加了兩道零知識證明（ZKP）：

1.  Paillier Blum 模數證明：保證公鑰 N 最多只能由兩個質數組成，防止被拆成一堆小數字。
    
2.  無小因子證明：確保公鑰 N 中絕對沒有小於 2 的 256 次方的小質數因子。
    

-   有了這兩道防線，駭客在第一步提交「偷工減料的保險箱」時就會被當場抓包並拒絕。
    

### 如何檢測歷史數據是否曾被攻擊？

-   檢測原理：駭客攻擊的鐵證是歷史數據中一定存在「包含小因子的公鑰 N」。
    
-   操作方式：使用成熟的大整數分解算法工具，批量檢查過去所有 MPC 參與方提交的公鑰 N。
    
-   後續建議：若檢測出小因子，說明該錢包可能已被攻擊，應立即將資產轉移到已修復漏洞的新錢包中。

# 5/18(一)課程筆記

### 核心觀念：AI 時代的開發者定位

-   **基礎知識的不可取代性**：有了 AI，基礎知識反而更重要。AI 能大幅提升 Coding 與學習效率，放大個人能力，但**系統複雜度、錯誤成本與安全邊界並沒有改變**。
    
-   **人的價值定位**：人負責設計架構，AI 協助細化與執行；人必須擔任 AI 產出的「審查者」。開發者若缺乏紮實的基礎知識與架構能力，將無法判斷 AI 生成的程式碼是否正確（安全的前提是基礎紮實，否則都是空中樓閣）。
    
-   **結論**：我們要駕馭 AI（將其作為放大器），而非被 AI 驅動，最終的審核與安全責任仍在開發者手中。
    

### Web2 與 Web3 的融合與差異

-   **並非互斥，而是共生**：Web3 不是完全獨立於 Web2。多數 Web3 專案（如交易所 Binance、預測市場 Polymarket）仍重度依賴 Web2 的前端頁面與後端管理技術，Web2 的經驗可大量複用。
    
-   **底層信任機制的轉換（以支付系統為例）**：
    
    -   **Web2 (Bank Transfer)**：透過 API 交互，信任基礎在於 KYC、密碼權限與「中心化機構後端 (Institutional Backend)」。
        
    -   **Web3 (穩定幣支付)**：將銀行機構替換為 Blockchain，不同公鏈形同不同銀行。信任基礎轉移為：透明度 (Transparency)、不可篡改性 (Immutability)、共識機制 (PoW, PoS) 以及私鑰簽名。
        

### Web3 核心命題：安全與資產保護

在 Web3 中，涉及大量資金交互，「安全」不是事後附加的插件，而是貫穿產品生命週期的核心屬性。在 Web3，**對安全的理解力等同於資產保護力**。

1.  錢包與身份
    

-   **定義**：錢包在 Web3 中代表用戶的「身份」與「資產管理」憑證，用以證明用戶行為。
    
-   **私鑰與簽名**：私鑰是透過數學算法保證的大數，運作邏輯為`Message -> Input Private Key -> Output Signature`。
    
-   **錢包分類與託管**：
    
    -   類別：EOA (公私鑰對)、Smart Contract Wallet (智能合約錢包，如 Safe 多簽)、AA (帳戶抽象，結合 EOA 與智能帳戶)。
        
    -   託管方式：自託管、全託管 (如交易所)、混合託管 (Hybrid)。
        
-   **錢包安全防線**：嚴防私鑰外洩 (Private Key Leak)、防範簽名欺騙 (如禁用 `eth_sign`、提高簽名可視性)、防止權限濫用。
    

2.Web3 交易週期與關鍵參數

交易週期：**構造 -> 簽名 -> 廣播**

-   **Gas (交易手續費)**`Gas Fee = Gas Count * Gas Price` (受 EIP-1559 影響)。
    
-   **Nonce (確保交易冪等性，防重放攻擊)**：
    
-   ETH (Account 模型)：使用預設 Nonce 排序。
    
-   Tron/Solana：無 Nonce，透過 SDK 使用交易 Hash 確保。
    
-   BTC (UTXO 模型)：Cash 只能用一次，原生具備冪等性。
    
-   **Calldata**：鏈上交易的靈魂，可視為 Interpreter，執行端會自行解析交易 Data  
    

3.交易安全實踐

-   **交易模擬** ：在提交交易前驗證執行結果，且**必須在「簽名」前進行模擬**。
    
-   **Server 端安全**：資金量拆分以減少單點風險；利用可信任執行環境 (TEE) 保護私鑰，確保無任何明文暴露。
