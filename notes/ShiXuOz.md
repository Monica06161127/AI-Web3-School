## AI基础部分

### 提示词

1.  关于提示词，提示词不是越长越好，而是需要给AI划定工作界面。是需要告诉模型：本次任务是什么，哪些信息可以使用，哪些输入只是参考，输出应该是什么格式，遇到不确定信息时应该怎么处理。
    
2.  一个实用写法是把 instruction 拆成四段：
    
    -   任务目标
        
    -   可用输入
        
    -   禁止行为
        
    -   输出格式和失败格式
        
3.  要注意提示词的安全问题
    

### 检索增强生成（RAG）

1.  RAG的主要作用是让AI生产的内容有来源，减少AI幻觉。同时也让AI模型训练的更加快速。
    
2.  一个 RAG 系统至少有三次判断：文档怎么切，查询时取哪些内容，生成时如何引用和拒答。任何一层做错，模型都会拿着错误材料说得很顺。
    
    -   文档切分（Chunking）的目的是让大语言模型（LLM）能够在有限的上下文窗口内，看到最相关的精确信息。切得太大，容易引入无关噪声；切得太小，又会丢失上下文语义。
        
    -   用户输入问题后，我们要从向量数据库中把“切好”的文档捞出来。取哪些、取多少，直接决定了回答的质量。
        
    -   拒答是 RAG 系统可靠性的底线。引用能极大提升系统的可信度。
        

### 智能体（Agent）

1.  模型负责提出候选行动，系统负责限制行动空间，用户负责批准高风险边界。
    
2.  一个 Agent 如果只有模型和工具，就像一个会说话的脚本执行器。真正可用的 Agent 必须知道自己能做什么、不能做什么、做完如何验证、失败如何停止、谁能审计它做过的事。
    
    -   **工具比回答更危险**：读数据、写数据库、发送请求、改配置不是同一风险等级。
        
    -   **状态必须外置和可查**：任务进度、工具结果、失败原因、用户确认都要记录，不应只藏在模型上下文里。
        
    -   **停止条件要明确**：达到目标、超出预算、信息不足、风险越界、用户拒绝，都应该让 Agent 停下来。
        
3.  对于Agent使用方面，需要：
    
    -   工具层进行权限控制。
        
    -   要有人类介入机制，进行人为审批确认。
        
    -   提示词与输入输出过滤，防止恶意提示词注入。
        
    -   Agent要生产所有行动日志，方便排查复盘。
        

### 模型上下文协议（MCP）

1.  模型不应该直接拥有世界；它应该通过明确协议访问被授权的上下文和工具。
    
2.  为AI制定标准协议和接口，方便在任何场景运行。
